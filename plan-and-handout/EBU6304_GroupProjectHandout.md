## 1. 基本信息

在接下来的几周内，您的团队将开发一个用于北京邮电大学国际学院招募助教的软件应用程序。敏捷方法应应用于开发的各个阶段，包括需求、分析与设计、实现和测试。迭代应仔细规划和记录，以展示增量交付、持续反馈和反思改进。

最终产品可以包含任何内容，本讲义中提供的规范仅包含高层级的抽象需求。需要认识到，确定软件需求是任何开发项目最关键和复杂的阶段之一。您应该应用适当的需求发现技术和敏捷实践，以合适的详细程度识别实际需求。最重要的是，您必须通过迭代对要实施的功能进行优先级排序，同时考虑可行性和客户需求。与真实世界软件开发一样，您应该明确且适当地定义项目范围。保持设计简单、模块化和可扩展。证明您的决策、流程、权衡和设计选择是本课程作业的重要组成部分。

## 2. 项目规范

### 2.1 基本要求

北京邮电大学国际学院每学期都会招募助教以支持学术模块和各种学院活动。目前，申请流程依赖表格和 Excel 文件。需要一个软件应用程序来简化工作流程。您将作为一个敏捷软件开发团队来开发一个简单的招聘系统。

该系统应满足北京邮电大学国际学院为模块负责人和其他活动（如监考）招募助教的特定需求。虽然部分任务要求您使用合适的技术分析招聘流程并确定需求，但以下是一些建议功能，以帮助您开始。建议的功能可包括但不限于以下内容：

**TA（助教申请者）功能：**

- TA 可以创建申请人档案
- TA 可以上传简历
- TA 可以查找可用职位
- TA 可以申请职位
- TA 可以查看申请状态

**MO（模块负责人）功能：**

- MO 可以发布职位
- MO 可以选择职位的申请人

**管理员功能：**

- 管理员可以查看 TA 的总体工作量

**人工智能驱动的功能（可选）：**

- 职位与申请人的技能匹配
- 识别申请人缺失的技能
- 平衡工作量

必须制作一个完整的应用程序原型。虽然不需要为原型中呈现的每个功能都实现完全功能的代码，但团队必须实现一组选定的核心功能。这些核心功能应展示系统的主要价值。

### 2.2 其他要求（强制性）

- 软件必须开发为以下两种形式之一：
  - a) 独立的 Java 应用程序，或
  - b) 轻量级的基于 Java Servlet/JSP 的 Web 应用程序。

- 所有输入和输出数据应存储在简单的文本文件格式中。例如纯文本 (.txt)、CSV、JSON 或 XML。**不要使用数据库**。

- 遵守上述限制对于确保以下几点至关重要：
  1. 重点仍放在基本的软件工程原理上，而非框架或语言的复杂性。引入高级工具，如 Spring Boot 或数据库集成，可能会将注意力转向框架特定的功能和配置，这不是本模块的主要学习目标。
  2. 系统保持与现有数据的兼容性，并能与遗留系统互操作。
  3. 对所有具备必要先决知识和技能的学生保持公平。

您的任务包括使用敏捷方法定义上述软件的详细需求、设计、开发和测试。您可以自由设计系统，只要它满足核心客户需求并且项目范围定义明确即可。

### 2.3 人工智能辅助开发

鼓励您在项目的各个方面使用 AI 提供帮助，包括头脑风暴、用户故事编写、原型设计、设计、编码、调试和测试。但是，您不应仅依赖 AI 生成的内容，因为 AI 工具有局限性。其输出可能缺乏特定领域的知识、包含细微的逻辑错误、产生低效的代码，或忽视伦理和隐私问题。此外，AI 解决方案通常是通用的，可能无法反映解决现实问题所需的批判性思维和适应性。为确保理解并避免过度依赖，您应准备好在现场演示期间回答问题，例如"为什么 AI 会提出这个建议？"或"您如何修改了 AI 的解决方案？"，以展示您的推理和修改。

如果您实现了 AI 驱动的功能，您必须以负责任和批判性的方式集成 AI。不能盲目接受 AI 生成的输出。您的系统必须将结构化逻辑与基于 AI 的解读相结合，并提供可解释的结果。

## 3. 敏捷管理与版本控制

每组有 6 名学生。您是与团队一起完成项目的敏捷团队。组内的所有学生都必须参与项目的所有方面，以获得全面的软件工程技能。您应该使用在讲座中学到的技术来管理项目，例如 Scrum、每日站会、围桌工作、Scrum Master 和决策等。

每个团队成员必须在 GitHub 上有为其分配的开发任务创建的可见分支和相应的提交记录。这些将作为个人贡献的关键证据。

每组将被分配一名助教，以提供支持、反馈、监控小组进展和个人贡献。您的助教应是您遇到问题或疑问时的首选联系人。助教将定期检查您小组的进展和个人贡献。助教不负责项目的评分。评分由软件工程教学团队的讲师进行。确保您的助教能够访问您的 GitHub 仓库。请参阅 QMPlus 上的助教分配文件，以了解分配给您的助教联系方式。

小组所有成员都应是主项目仓库的协作者。主项目仓库（即主分支）必须可被分配的助教访问。助教在必要时可请求访问子分支。

必须明确使用 GitHub 来展示您的贡献，通过诸如提交、分支、拉取请求、议题和 README 更新等活动。每个成员都必须积极参与，以展示在团队中的明确参与度和贡献。在评分过程中，GitHub 上证据不足将被视为贡献不足，没有例外。

## 5. 分阶段评估

本次课程作业将通过三个评估点进行管理和评估。

### 5.1 第一次评估 - 30%

这包括产品待办事项列表、原型和一份简短报告。

**QMPlus 提交：**

- 产品待办事项列表，一个 Excel 文件（QM+ 上提供了模板）。
  - 文件名：ProductBacklog_groupXXX.xlsx，其中 XXX 是您的组号。它应包含所有带有验收标准、优先级、估算和迭代计划的用户故事。
- 原型，一个 PDF 文件。
  - 文件名：Prototype_groupXXX.pdf，其中 XXX 是您的组号。它应包含完成后的原型。仅需要低保真度或中保真度原型。
- 简短报告，一个 PDF 文件（QM+ 上提供了模板）。
  - 文件名：Report_groupXXX.pdf，其中 XXX 是您的组号。
  - 本文件应描述项目中使用的需求发现技术、迭代计划、优先级排序和估算方法（最多 5 页）。还应包括支持材料（不计入 5 页限制内），例如访谈问题、调查结果和客户反馈（如适用）。

将所有以上文件制作成一个 .ZIP 压缩文件并在 QM+ 上提交。只有组长应代表全组提交文件。

**GitHub：** 将检查您的 GitHub 仓库。确保其已正确设置并清晰地显示成员的提交记录。

### 5.2 中期评估 - 20%

这涉及检查软件的中间版本。

**演示与答辩：** 这模拟了现实场景中与客户的会议，您向客户通报您的进展。所有小组成员都需要参加演示和答辩环节。您需要展示进展并演示最新版本的软件（理想情况下，每个成员发言 2 分钟）。在答辩期间，每个成员将被问 1 到 2 个问题。

**GitHub：** 将检查您的 GitHub 仓库，以获取软件的最新版本和每个成员的贡献。确保在检查日期前将所有子分支的贡献合并到主分支。

### 5.3 最终评估 - 50%

这涉及检查最终软件、演示和报告。

**QMPlus 提交：**

- 简短报告，一个 PDF 文件（QM+ 上提供了模板）。
  - 文件名：Report_groupXXX.pdf，其中 XXX 是您的组号。
  - 本报告记录了软件设计策略、构建计划、测试策略和技术、生成式人工智能的使用等。它应包含小组报告部分（最多 10 页）和个人陈述部分（每人不超过 300 字）。更多详细信息可在模板中找到。
- 软件，一个 ZIP 文件。
  - 文件名：Software_groupXXX.zip，其中 XXX 是您的组号。
  - 它应包括以下内容：
    - 源代码。
    - 测试程序。
    - 代码文档，例如 JavaDocs。
    - 用户手册，包括应用程序的主要界面截图（每个主界面至少一张）。
    - 一个 readme 文件，提供设置、配置和运行软件的说明。
- 演示视频，录制一个短视频来演示您的软件，并将其提交到 QM+。演示视频的要求：
  - 可以使用任何屏幕录制软件制作。
  - 不超过 10 分钟。
  - 您应该像演示销售产品一样展示软件的运行情况。您应该演示所有功能的操作。建议也演示错误处理。
  - 您必须在视频中解释您的操作，必须录音，并且叙述必须使用英语。
  - 视频格式不重要，因为您将上传到 QM+ Media。
  - 视频不得进行后期编辑，且不得包含任何 AI 生成的内容。

**演示与答辩：** 这模拟了现实场景中与客户的最终会议，您交付最终产品并希望获得用户满意度。所有小组成员都需要参加演示和答辩环节。您将被要求完成验收测试任务，以验证软件是否满足指定的需求。在答辩期间，每个成员还将被问 1 到 2 个问题。

**GitHub：** 还将检查您的 GitHub 仓库，以获取软件的最新版本和每个成员的贡献。确保在检查日期前将所有子分支的贡献合并到主分支。
